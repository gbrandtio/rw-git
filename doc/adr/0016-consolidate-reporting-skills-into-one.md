# 0016 — Consolidate the five reporting skills/prompts into one

- **Status**: Accepted
- **Date**: 2026-07-06
- **Deciders**: rw_git maintainers
- **Related**: [ADR-0005](0005-server-side-interpretation-and-report-meta-tools.md)
  (the report meta-tools the skills teach),
  [ADR-0015](0015-report-tool-source-of-truth-and-structured-hints.md)
  (the `reportToolSources` map that generates deep-dive tool lists)

## Context

The `.agents/skills/` directory shipped five reporting skills
(`rw-git-mcp-reporting`, `-technical-reporting`, `-pm-reporting`,
`-security-reporting`, `-code-review-reporting`), each generated by
`tool/sync_prompts.dart` from a `SKILL.template.md` and each also emitted
as a native MCP prompt. Comparing the five templates line by line showed
they were one workflow parameterized on five axes: a persona title, which
`generate_*` tool to call, minor wording in the format rules, an optional
`checkout_branch` line in the prepare step, and which `reportToolSources`
key fed the `<deep_dive>` list. The top-level `rw-git-mcp-reporting` skill
was already a superset whose deep-dive tool list was the union of the
other four and whose report step explicitly handed off to them.

Five near-identical surfaces meant five places for wording to drift, five
prompts for an MCP client to choose between (against ADR-0007's tools-list
token-budget spirit), and a maintenance rule ("update five templates in
lockstep") that ADR-0015 had already had to repair once.

## Decision

- One canonical template, `.agents/skills/rw-git-mcp-reporting/
  SKILL.template.md`, replaces all five. It carries a `<report_selection>`
  goal-to-tool decision table instead of a persona per report type, one
  generalized `<format_requirements>` block, and a `<deep_dive>` section
  with one generated tool list per report type — five
  `<!-- generate:deep_dive_tools report=... -->` markers, one per
  `reportToolSources` key, so per-report deep-dive granularity is kept
  without duplicating any prose. The codegen (`expandGenerated`,
  `renderDeepDiveTools`) needed no changes; only `promptSkillNames`
  shrinks to the single entry.
- The `checkout_branch` guidance moves into the shared
  `_shared/reporting_prepare_step.md` partial (it applies whenever the
  goal targets a branch), removing the last per-template duplication.
- The four specialized skill directories, their generated prompt Dart
  files, and their `registerPrompt` calls are deleted.
  `rw-git-mcp-installation` is untouched: it is a human setup guide with
  no shared structure, and it already sat outside `promptSkillNames`.
- The npm distribution needs no change: its `prepack` script copies
  `.agents/skills` and `README.md` wholesale, so the consolidation
  propagates at the next publish.

## Consequences

- **Positive**: one template, one generated skill, one generated prompt —
  a wording or contract change lands in exactly one place, and
  `prompts/list` shrinks by four entries.
- **Positive**: the drift tests (`test/mcp/prompts_sync_test.dart`) now
  assert the single skill carries a generated deep-dive list for *every*
  `reportToolSources` key, so a new report type cannot ship without its
  deep-dive routing.
- **Negative / breaking**: MCP clients pinned to a removed prompt name by
  string break; documented in `CHANGELOG.md` under `3.2.0`. The
  consolidated prompt is the migration target for all of them.
- **Trade-off**: the single prompt is longer than any one specialized
  prompt was. Accepted: a model reads one selection table instead of the
  client choosing among five prompts, and the per-report deep-dive lists
  remain individually scoped, so the token cost of acting on any single
  goal is unchanged.
